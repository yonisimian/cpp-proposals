## Abstract

This proposal introduces an extension to C++'s structured bindings, allowing for the assignment of values to existing variables.

## Motivation

Allowing assignment to existing variables within structured bindings addresses a practical gap in the language. The key motivations are:

1. **Standard Library Independence:** Environments where the standard library is unavailable (embedded systems, bare-metal development, OS kernels) cannot use `std::tie`. A language-level feature provides this capability universally.

2. **Reduced Cognitive Load:** Structured bindings are intuitive and widely taught. Extending them to support existing variables is more natural than requiring developers to learn `std::tie` as a separate concept.

3. **Unified Syntax:** Currently, `std::tie` handles existing variables while structured bindings handle new declarations. Allowing both in a single syntax improves code clarity and reduces the need for multiple constructs.

4. **Common Pattern Support:** Reassigning multiple values in loops or conditional branches is a frequent pattern. The current workaround requires mixing structured bindings with `std::tie`, leading to inconsistent code.

## Introduction

C++17 introduced structured bindings, a powerful language feature that allows for the decomposition of a tuple-like object into its constituent elements. The syntax `auto [x, y] = foo();` is a significant improvement in readability and conciseness over traditional methods.

However, a core limitation of structured bindings is their inability to assign to pre-existing variables. The current approach requires the use of `std::tie` from the `<tuple>` header, as shown below:

```cpp
#include <tuple>

std::tuple<Token, Value> scan(); // scans the next line on each invocation

for (auto [token, value] = scan(); token != EOF; std::tie(token, value) = scan()) {
    // ... parse token
}
```

While effective, `std::tie` is not a language-level feature. Its use presents three main challenges:

1. **Standard Library Dependency:** `std::tie` is part of the standard library. Many developers, particularly in embedded systems, work on platforms where the standard library is either unavailable or has a significant footprint, making its use unfeasible. The core C++ language could provide a solution for this common problem.

2. **Cognitive Load:** Structured bindings are highly intuitive and now a core part of C++ pedagogy. Providing a language-level alternative to std::tie reduces the amount of specialized knowledge required for common programming tasks.

3. **Only existing variables:** `std::tie` can only be used with existing variables, while structured bindings can only declare new ones. A unified syntax that allows both would enhance code clarity and reduce the need for multiple constructs to achieve similar goals.

This proposal aims to address these issues by extending structured binding syntax to support assignment to existing variables.

## Proposal

Extend structured binding syntax to optionally use the `using` keyword for elements in the _sb-identifier-list_ inside the binding list to denote a binding to an existing variable.

The main difference from current syntax is the ability to use `using` before a variable name in the structured binding list to indicate that the variable is already declared and should be assigned to and not initialized.

### Example

```cpp
int x;
auto [using x, y] = get_pair(); // y is a new variable, x is assigned to
```

### Explanation

`using` is already a keyword. By allowing it inside the structured binding list, we can extend this meaning to "using an existing variable in the scope." This approach is intuitive even in the English languge level.

When an existing variable is prefixed with `using`, implementation should find the variable in the current scope and assign the corresponding element of the tuple-like object to it using `operator=`. The `auto` keyword at the beginning of the statement signals that this is a structured binding operation.

## Motivation and Use Cases

### 1. Embedded and Bare-Metal Systems

The most compelling technical motivation for this proposal is its applicability in **constrained environments**. In projects for embedded systems, game consoles, or operating system kernels, a full C++ standard library is often unavailable. These environments lack headers like `<tuple>`, making `std::tie` inaccessible. A language-level feature provides a uniform, built-in solution that is always available.

### 2. Readability and Consistency

Structured bindings are a well-loved feature for their intuitive nature. `std::tie` requires a separate header, a different syntax, and introduces a new concept (a function that returns a `tuple` of references) to solve a common problem. The proposed syntax leverages an existing, familiar pattern, reducing cognitive overhead and making code more consistent and easier to understand for newcomers and seasoned developers alike.

### 3. Loop Reassignment Patterns

A common pattern is reassigning values in a loop, particularly with functions that return multiple values:

```cpp
// Current approach - mixing constructs
for (auto [token, value] = scan(); token != EOF; std::tie(token, value) = scan()) {
    process(token, value);
}

// With this proposal - consistent syntax
for (auto [token, value] = scan(); token != EOF; auto [&token, &value] = scan()) {
    process(token, value);
}
```

### 4. Retry and Recovery Patterns

When a function returns a status alongside a value, retry logic often needs to reassign:

```cpp
auto [result, status] = try_operation();
while (status == Status::Retry) {
    prepare_retry();
    auto [&result, &status] = try_operation();  // reassign both
}
```

### 5. Partial Updates

Sometimes only some values need to be reassigned while introducing new bindings:
```cpp
int id;
auto [&id, name] = get_primary_record();
// ... use id and name ...
auto [&id, address] = get_secondary_record();  // update id, get new address
```

## Interaction with C++26 Placeholder `_`
C++26 introduces `_` as a placeholder for unused variables in structured bindings. This proposal is designed to work seamlessly with that feature:

```cpp
int x;
auto [&x, _] = get_pair();  // assign first element to x, discard second
```

This combination provides a complete replacement for `std::tie` with `std::ignore`:

| `std::tie` with `std::ignore` | With P3817 and C++26 |
|-------------------------------|----------------------|
| `std::tie(x, std::ignore) = get_pair();` | `auto [&x, _] = get_pair();` |
| `std::tie(std::ignore, y) = get_pair();` | `auto [_, &y] = get_pair();` |

Note that `_` is a language feature in C++26 and does not require any library headers, making it suitable for embedded environments.

## Alternative Syntaxes Considered

This proposal presents two primary syntax alternatives for the committee's consideration. Both achieve the same semantic goal of allowing assignment to existing variables within structured bindings.

### Option A: `&` Symbol

```cpp
int x;
auto [&x, y] = get_values();  // x is assigned, y is declared
```

**Pros:**
- Concise and lightweight syntax
- Leverages existing C++ semantics where `&` denotes "referencing something that exists"
- Familiar to developers already comfortable with reference syntax

**Cons:**
- Potential ambiguity with function pointers when used with expressions like `&get_reference()` (see [Returned References](#returned-references))
- Visual similarity to `auto& [x, y]` may cause initial confusion
- The combination `]]` in expressions like `auto [&map[key], y]` resembles attribute syntax

### Option B: `using` Keyword

```cpp
int x;
auto [using x, y] = get_values();  // x is assigned, y is declared
```

**Pros:**
- Unambiguous parsing - no confusion with address-of operator or function pointers
- Clear semantic intent: "using" an existing variable rather than declaring a new one
- Works cleanly with returned references: `auto [using get_reference(), y] = get_values();`
- Mirrors the concept of `using` declarations bringing names into scope

**Cons:**
- More verbose than the `&` syntax
- Introduces contextual keyword usage within structured bindings

### Other Syntaxes Considered (Not Recommended)

#### `=` Symbol

```cpp
int x;
auto [=x, y] = get_values();
```

Not recommended due to confusion with lambda capture syntax where `[=]` denotes capture-by-value.

#### `let` Keyword

```cpp
int x;
let [&x, y] = get_values();
```

Not recommended due to potential ambiguity with Pattern Matching proposals.

#### `tie` Keyword

```cpp
int x;
auto [tie x, tie y] = get_values();
```

Considered for familiarity with `std::tie`, but rejected as overly verbose and potentially confusing since this is not actually using `std::tie`.

#### No Specifier (Implicit)

```cpp
int x;
auto [x, y] = get_values();  // x would be assigned if it exists
```

Not recommended as it would silently change behavior based on whether a variable exists in scope, violating the principle of least surprise and making code harder to reason about.

## Implementation Notes

This is a language extension, not a standard library addition. A compiler would need to implement this new parsing rule. The implementation would involve checking the variables in the `binding-list` against the variables in the current scope. If an existing variable is found with the `assigns` keyword, the compiler would generate code to perform an assignment using the variable's `operator=`. If `auto` is used, a new variable would be declared.

## Conclusion

This proposal for structured bindings for existing variables offers a clean, consistent, and powerful language feature. It solves a real-world problem for developers in constrained environments and provides a more intuitive syntax for all C++ programmers. The proposed syntax is explicit, avoids keyword ambiguity, and aligns with the modern direction of the C++ language.

The working group is encouraged to discuss this proposal and provide feedback on the proposed syntax and rationale.
