## Abstract

This proposal introduces an extension to C++'s structured bindings, allowing for the assignment of values to existing variables.

## Motivation

Structured bindings (`auto [x, y] = foo();`) can only declare new variables. Assigning to pre-existing variables requires `std::tie` from `<tuple>`. There is no single construct that does both.

This proposal closes that gap and adds additional advantages:

1. **Mixed-mode** — Neither `std::tie` nor structured bindings alone allow some elements to be new variables and others pre-existing in the same statement. P3817 uniquely enables this:
   ```cpp
   int id;
   auto [using id, name] = get_record();  // id is assigned; name is declared
   ```

2. **Unified syntax** — One construct for both assignment and initialization reduces cognitive overhead and produces more consistent code. Currently initializing uses structured bindings; reassigning uses `std::tie`. 

3. **No standard library required** — `std::tie` requires `<tuple>`, unavailable in many constrained environments (embedded systems, bare-metal, OS kernels). A language-level feature works everywhere C++ does.

4. **Encouraged by P0144R2** — The authors of the original structured bindings paper explicitly invited this extension [section 3.3]:
   > "This can always be proposed separately later as a pure extension if desired."

## Proposal

Extend structured binding syntax to allow the `using` keyword before an identifier in the _sb-identifier-list_ to denote assignment to an already-declared variable, rather than declaration of a new one.

### Syntax

The grammar change is purely additive — `sb-identifier` is extended to optionally accept `using`:

```
sb-identifier:
    ...opt usingopt identifier attribute-specifier-seqopt
```

When `using` prefixes an identifier, the implementation locates the variable in the current scope and assigns the corresponding element via `operator=`. Without `using`, the existing declaration semantics apply unchanged.

```cpp
int x;
auto [using x, y] = get_pair();  // x is assigned; y is a new variable
```

### Semantics

The hidden variable `e` is introduced exactly as today (per [[dcl.struct.bind] ¶1](https://eel.is/c++draft/dcl.struct.bind)). The difference is in how each element SBᵢ is resolved: for a `using`-marked element, instead of introducing a new name, the implementation emits an assignment to the existing variable.

The three cases follow the structure of [[dcl.struct.bind]](https://eel.is/c++draft/dcl.struct.bind):

#### Arrays

**Given**
```cpp
int ar[2];
int x;
```

**Today:**
```cpp
auto [e1, e2] = ar;
```
expands to:
```cpp
int __e_today[2] = {ar[0], ar[1]};
int& e1 = __e_today[0];
int& e2 = __e_today[1];
```

**With P3817R0:**
```cpp
auto [using x, y] = ar;
```
expands to:
```cpp
int __e_p3817[2] = {ar[0], ar[1]};
x = std::move(__e_p3817[0]);  // assigned via move
int& y = __e_p3817[1];        // new binding
```

#### Tuple-like

##### Regular tuple-like types

**Given**
```cpp
std::pair<int, int> get_pair();
int x;
```

**Today:**
```cpp
auto [e1, e2] = get_pair();
```
expands to:
```cpp
std::pair<int,int> __e_today = get_pair();
int&& e1 = std::get<0>(static_cast<std::pair<int,int>&&>(__e_today));
int&& e2 = std::get<1>(static_cast<std::pair<int,int>&&>(__e_today));
```

**With P3817R0:**
```cpp
auto [using x, y] = get_pair();
```
expands to:
```cpp
std::pair<int,int> __e_p3817 = get_pair();
x = std::get<0>(static_cast<std::pair<int,int>&&>(__e_p3817));
int&& y = std::get<1>(static_cast<std::pair<int,int>&&>(__e_p3817));
```

##### Custom tuple-like types

For types whose move assignment has observable side effects, P3817 introduces a behavioral difference from today: the move happens **immediately at the binding statement**, not deferred to the point of use.

**Given:**
```cpp
struct Loud {
    int value;
    Loud& operator=(Loud&& other) {
        std::println("moved");
        value = other.value;
        return *this;
    }
};

std::pair<Loud, Loud> get_pair();
Loud x;
```

**Today:**
```cpp
auto [e1, e2] = get_pair();
std::println("after binding");
Loud a;
a = std::move(e1);        // move happens here, not at binding
std::println("after use");
```
Output:
```
after binding
moved
after use
```

**With P3817R0:**
```cpp
auto [using x, y] = get_pair();
std::println("after binding");
```
Output:
```
moved
after binding
```

The move happens at the binding statement itself, before any subsequent code — including code that might not expect it yet.

### Interactions

#### `auto&` — Ill-Formed

A structured binding declaration with a ref-qualifier combined with a `using`-marked element is ill-formed:

```cpp
auto& [using x, y] = ar;  // ill-formed
```

The outer ref-qualifier and the assignment semantics of `using` are contradictory: the ref-qualifier binds `e` by reference (no copy is made), but assigning to `x` requires a writable copy to move from.

#### `const` — Ill-Formed for Assigned Elements

```cpp
const auto [using x, y] = ar;  // ill-formed: cannot assign in a const binding
```

#### Returned Lvalues

The `using` specifier may also appear before an expression yielding an lvalue, not just a plain identifier. This is the language-level equivalent of `std::tie` with non-variable arguments:

```cpp
// std::tie equivalent
std::tie(foo(), s[0]) = get_pair();

// with P3817
auto [using foo(), using s[0]] = get_pair();
```

There is no ambiguity: taking the address of a named function is not meaningful in a structured binding context, so `using foo()` unambiguously means "assign to the lvalue returned by `foo()`".

#### C++26 `_` Placeholder

P3817 composes naturally with C++26's `_` placeholder for discarding elements:

```cpp
int x;
auto [using x, _] = get_pair();  // assign first element to x, discard second
```

This provides a complete, library-free replacement for `std::tie` with `std::ignore`:

| `std::tie` with `std::ignore` | With P3817 and C++26 |
|-------------------------------|----------------------|
| `std::tie(x, std::ignore) = get_pair();` | `auto [using x, _] = get_pair();` |
| `std::tie(std::ignore, y) = get_pair();` | `auto [_, using y] = get_pair();` |

## Alternative Syntaxes Considered

### Option A: `&` Symbol

```cpp
int x;
auto [&x, y] = get_values();
```

**Pros:**
- Concise
- `&` suggests "referencing something that already exists"
- Familiar to developers comfortable with reference syntax

**Cons:**
- Ambiguous with function pointers in expressions like `auto [&get_reference(), y] = ...`
- `]]` in `auto [&map[key], y]` resembles attribute syntax
- Visual similarity to `auto& [x, y]` may cause initial confusion

### Option B: `using` Keyword *(preferred by authors)*

```cpp
int x;
auto [using x, y] = get_values();
```

**Pros:**
- Unambiguous — no confusion with address-of or function pointers
- Clear semantic intent: "using" an existing variable rather than declaring a new one
- Works cleanly with returned lvalues: `auto [using get_reference(), y] = ...`
- Mirrors the concept of `using` declarations bringing names into scope

**Cons:**
- More verbose than `&`
- Introduces contextual keyword usage within structured bindings

### Other Syntaxes (Rejected)

- **`=x`** — Conflicts with lambda capture-by-value intuition (`[=]`).
- **`let`** — Conflicts with Pattern Matching proposals.
- **`tie x`** — Overly verbose; confusingly evokes `std::tie` even though it is not.
- **Implicit (no specifier)** — Would silently assign or shadow based on scope, violating the principle of least surprise.
